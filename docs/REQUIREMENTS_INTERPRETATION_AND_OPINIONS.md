# 요구사항 해석 및 개선 의견

과제 요구사항을 어떻게 해석했는지, 구현 시 세운 가정, 그리고 추가로 개선하고 싶은 점을 정리합니다.

> 비동기 처리 구조·재시도 정책의 기술 설명은 [`ASYNC_AND_RETRY.md`](./ASYNC_AND_RETRY.md)를 참고하세요.

---

## 요구사항 해석

### 핵심 해석 요약

| 과제 요구 | 해석·구현 방향 |
|-----------|----------------|
| 즉시 발송 아님 | API는 발송 완료가 아니라 **접수(큐잉)**. POST → 202 + DB에 **PENDING** 저장, 실제 발송은 Worker |
| 비즈니스 TX와 알림 분리 | 수강 등 본업무 **커밋 이후**에만 알림 생성. `AFTER_COMMIT` 이벤트 + 알림 생성은 **별도 TX(`REQUIRES_NEW`)** |
| 재시도·DEAD | 발송 실패 시 **FAILED**, `nextRetryAt`·`failureReason` 기록, 최대 3회 재시도 후 **DEAD** |
| 중복 방지 | 동일 (수신자, 타입, referenceId, 채널)은 1건만. UK + 사전 조회, 동시 요청은 **409** |
| MQ 없이 운영 가능 | 메시지 브로커 대신 **DB 상태 + 폴링**(Outbox 유사). 이후 Kafka Consumer 등으로 교체 가능하도록 설계 |
| 다중 인스턴스·운영 | `SKIP LOCKED`로 claim, PROCESSING **스턱 복구**, 재시작 후 DB에 남은 알림 재처리 |
| 선택: 예약 발송 | `scheduledAt` null = 즉시, 미래 시각 = claim 대상에서 제외 |
| 선택: 읽음 | IN_APP `PATCH /read`, 멱등 처리 |
| 선택: DEAD 수동 재시도 | `POST /retry`로 DEAD → PENDING, `retryCount` 리셋 |

### 항목별 해석 (왜 이렇게 이해했는지)

**비동기 발송**  
요구사항의 “즉시 발송하지 않는다”는 것은 API 응답 시점에 사용자에게 도달했다는 뜻이 아니라, **발송 요청을 시스템에 맡기고 백그라운드에서 처리**한다는 의미로 받아들였습니다. 그래서 API 스레드와 발송 I/O를 분리하고, 상태(PENDING → PROCESSING → SUCCESS/FAILED/DEAD)로 진행 상황을 추적할 수 있게 했습니다.

**비즈니스 트랜잭션 분리**  
수강신청이 성공했는데 알림 INSERT 실패로 수강까지 롤백되면 안 된다고 판단했습니다. 반대로, 수강이 롤백됐는데 알림만 남는 것도 피해야 하므로, 알림 생성은 **본업무 커밋 이후**에만 수행하도록 `AFTER_COMMIT` 이벤트를 사용했습니다.

**재시도·DEAD**  
외부 발송(이메일 등)은 일시적 장애가 흔하므로, 실패를 끝이 아니라 **재시도 가능한 상태(FAILED)** 로 두고, 횟수·시간 정책을 두어 무한 재시도를 막았습니다. 한계를 넘으면 **DEAD**로 두어 운영자가 수동 재시도(`POST /retry`)할 수 있게 했습니다.

**중복 방지**  
동일 수강 확정 이벤트가 네트워크 재전송·더블 클릭 등으로 두 번 들어와도 알림은 한 번만 가야 한다고 해석했습니다. DB 유니크 키와 사전 exists 검사로 **동일 이벤트·채널 1건**을 보장합니다.

**MQ 없이 운영**  
과제 범위에서 브로커 설치 없이도 동작해야 하므로, 알림 row를 **작업 큐**처럼 두고 스케줄러가 폴링·선점하는 방식으로 구현했습니다. 장기적으로는 Outbox + Kafka로 발송 단계만 분리할 수 있다고 봤습니다.

**운영(스턱·다중 인스턴스)**  
Worker가 죽으면 PROCESSING에 멈출 수 있으므로, 일정 시간 이상이면 스턱으로 보고 복구하는 요구로 이해했습니다. 여러 인스턴스가 동시에 같은 row를 잡지 않도록 `SKIP LOCKED` 선점을 적용했습니다.

**선택 기능**  
예약 발송은 “아직 보낼 시간이 아님”을 `scheduledAt`으로 표현하고, 읽음·DEAD 재시도는 운영·UX 보완으로 선택 구현했습니다.

### 구현 시 가정

- **User(id=1)** 을 Postman 등 수동 테스트용 수신자로 둔다 (기동 시 자동 생성).
- **referenceId** 는 강의 엔티티 없이 문자열(예: `course-1`)로 관리한다.
- **실제 EMAIL/SMS/푸시** 는 보내지 않고 `LoggingNotificationSender`로 로그 Mock 한다.
- **수강(enrollment)** 만 비즈니스 API로 구현하고, 강의·결제 알림 타입은 enum·이벤트 패턴으로 확장 가능하다고 본다.
- **인증/권한** 은 과제 범위外 — `receiverId`는 요청 body/query로 전달한다.

## 개선하고 싶은 점

| 항목 | 내용 |
|------|------|
| 알림 ID 응답 | 수강신청 응답에 `notificationId` 포함 (지금은 목록 API를 다시 호출해야 함) |
| 예약 발송 | 수강신청 API에도 `scheduledAt` 선택 입력 |
| 시간 기준 | KST 등 시간대를 API 문서에 명시 |
| 운영 숫자(메트릭) | 성공/실패/DEAD 건수를 숫자로 모아 장애를 빨리 파악 |
| 재시도 정교화 | 일시 오류만 재시도, 간격 점점 증가, 복구 불가 오류는 바로 DEAD (미구현 아이디어) |
